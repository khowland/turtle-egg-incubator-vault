## CR-1: Sidebar User/Version Display Repositioning & Simplification

CURRENT STATE:
- User name displayed as large `### 👤 {name}` header at top of sidebar
- Session ID displayed as caption below the name
- "SHIFT END" button below that
- Version stamp (`Clinical Standard v8.1.1`) shown as caption BELOW the
  divider at the very bottom of the sidebar — often scrolled off-screen
  or below the visible area on smaller displays
- Version also appears on: Login splash screen, Diagnostic page footer,
  Help page footer

PROBLEM:
- Version stamp at bottom of sidebar falls below visible screen area
- User identity and version are separated by the full navigation menu
- Session ID is technical noise for the end user
- Inconsistent placement across pages (some have footers, some don't)

REQUESTED CHANGE:
- Move user/observer display and version stamp to the TOP of the sidebar,
  above the navigation links
- Simplify to a single compact block:
    {observer_name}
    {VERSION}          (smaller muted font)
- Remove Session ID display from the sidebar (keep it in system_log only)
- Keep the SHIFT END button but move it to the bottom of the sidebar
- Remove redundant version captions from individual page footers
  (Diagnostic, Help) since it will always be visible in the sidebar

FILES AFFECTED:
  1. utils/bootstrap.py   — render_custom_sidebar() function (lines 174-201)
  2. vault_views/7_Diagnostic.py — footer caption (line 84)
  3. vault_views/8_Help.py — footer caption (line 144)
  4. utils/session.py — Login splash screen version display (line 63)

BEST PRACTICE NOTES:
- Streamlit sidebar renders top-to-bottom; placing identity at top ensures
  it's always visible regardless of menu length or screen size
- Industry standard: user identity + app version in sidebar header area
  (similar to Jira, GitHub, Notion sidebar patterns)
- The SHIFT END button should remain near bottom for safety (prevents
  accidental clicks when navigating)


## CR-2: Standardize Audit Log Vocabulary

CURRENT STATE (inconsistent phrasing):
  - "Biologist {name} clocked in."                (session.py:186)
  - "Biologist {name} explicitly ended shift."     (bootstrap.py:191)
  - "Session Active: {name} is here."              (1_Dashboard.py:35)
  - "SUCCESS: {event_type} (Entries: N)"           (audit.py:34)
  - "FAILURE: {event_type} - {error}"              (audit.py:42)
  - "CRITICAL: Incomplete Intake for Case {case}. Chain broke: {error}"
                                                    (2_New_Intake.py:342)
  - "Clinical Correction: Observation {id} voided. Reason: {reason}"
                                                    (3_Observations.py:474)
  - "S6 Rollback: Subject {id} reverted from Hatched. Ledger entries voided."
                                                    (3_Observations.py:518)
  - "Surgical Resurrection: Observation {id} restored."
                                                    (3_Observations.py:554)
  - "Forensic Export: Observer {name} downloaded egg CSV."
                                                    (6_Reports.py:465)
  - Various emoji-prefixed logger messages throughout

PROBLEM:
- Mixed vocabulary: "Biologist", "Observer", "Session Active", "is here"
- Non-standard phrasing: "clocked in", "explicitly ended shift"
- Inconsistent structure: some messages are sentences, some are fragments
- "Surgical Resurrection" and "BIO-RECOVERY" are jargon not useful for
  auditing or incident response

REQUESTED CHANGE:
Standardize ALL system_log event_messages to the following format:

  Pattern: "{action_verb}: {subject} — {details}"

  Event Type    | Standard Message Format
  --------------|----------------------------------------------------
  ACCESS        | "Session started: {user_name}"
  TERMINATE     | "Session ended: {user_name}"
  AUDIT         | "Data committed: {operation} ({count} records)"
  ERROR         | "Operation failed: {operation} — {error_summary}"
  EXPORT        | "Data exported: {format} by {user_name} ({details})"
  VOID          | "Record voided: {record_type} {id} — Reason: {reason}"
  ROLLBACK      | "State reverted: {subject} {id} — {details}"
  RESTORE       | "Record restored: {record_type} {id}"
  INTAKE_ERROR  | "Intake failed: Case {case_number} — {error_summary}"

  Key vocabulary standardization:
  - "Biologist" → "user" or simply use the observer_name
  - "clocked in" → "Session started"
  - "explicitly ended shift" → "Session ended"
  - "is here" → remove (redundant with session start)
  - "Surgical Resurrection" → "Record restored"
  - "BIO-RECOVERY" → "Session resumed" (in st.success messages)
  - "Chain broke" → "Transaction failed"

FILES AFFECTED:
  1. utils/session.py        — lines 151, 186 (login messages)
  2. utils/bootstrap.py      — line 191 (shift end message)
  3. utils/audit.py          — lines 34, 42 (generic audit wrapper)
  4. vault_views/1_Dashboard.py — line 35 (session active message)
  5. vault_views/2_New_Intake.py — line 342 (intake error)
  6. vault_views/3_Observations.py — lines 474, 518, 554 (void/rollback/restore)
  7. vault_views/6_Reports.py — lines 282, 292, 465 (export messages)

BEST PRACTICE NOTES:
- NIST SP 800-92 (Guide to Computer Security Log Management) recommends:
  structured, consistent, machine-parseable log messages
- Pattern "Action: Subject — Details" is grep-friendly and sort-friendly
- Avoid colloquial language in audit logs ("clocked in", "is here")
- Avoid emojis in database-stored audit messages (keep emojis for UI only)
